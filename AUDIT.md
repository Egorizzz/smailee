# Smailee — Аудит кода, безопасности и сценариев

> Рабочий документ аудита. Статус: в работе → по мере исправления пункты помечаются ✅.

## 1. План аудита

1. **Код-ревью**: логические ошибки, конфликты, недоработки движков.
2. **Безопасность**: публичные endpoint'ы, авторизация, утечки, XSS, секреты.
3. **Сценарии** (пользователь + админ): полный список, проверка реализуемости,
   достройка недостающего (тарифы, оплата, админка, оферта).
4. **CI/CD**: процесс, при котором сервис не падает от плохого деплоя.

| # | Проблема | Серьёзность | Решение |
|---|----------|-------------|---------|
| R1 | `/api/inbound` без аутентификации — любой может слать фейковые «ответы клиентов», триггерить AI и создание лидов | КРИТИЧНО | Секрет `INBOUND_SECRET` в заголовке |
| R2 | `/api/leads/export` доступен ЛЮБОМУ залогиненному юзеру — все заявки с лендинга утекают любому клиенту | КРИТИЧНО | Только для роли ADMIN |

| R5 | Лид-форма `/api/leads` без rate-limit — спам заявками | СРЕДНЕ | Простой in-memory rate limiter по IP |

| R8 | `checkWarmupLimit` — гонки при параллельной отправке (не атомарный инкремент) | НИЗКО | Приемлемо: worker однопоточный, batch последовательный |
| R9 | Открытая регистрация — любой может создать аккаунт с полным функционалом | ВЫСОКО (бизнес) | Тарифная модель: новый аккаунт = TRIAL с лимитами; полный функционал после оплаты |
| R10 | `simulateReply` — оставлен намеренно (демо-инструмент), доступен только владельцу кампании | ОК | — |

## 3. Безопасность — что уже хорошо

- Пароли: bcrypt (10 раундов). Сессии: JWT в httpOnly+sameSite cookie, secure в проде.
- Server Actions Next.js — встроенная CSRF-защита (Origin check).
- Prisma ORM — нет SQL-инъекций. Zod-валидация входных данных лид-формы.
- Секреты только в `.env` (в gitignore), в коде — `process.env`.
- Мультиарендность: все запросы ЛК фильтруются по `userId` (проверено по коду).
- Трекинг-endpoint'ы (open/click/unsubscribe) публичные **by design** (вызываются из письма).

## 5. Что достраиваем (реализация в этом аудите)

1. **Роль ADMIN** (`User.role`), первый админ назначается через env `ADMIN_EMAIL`.
2. **Тарифы**: `User.plan` (TRIAL/START/PRO) + `planExpiresAt`. Лимиты в `src/lib/plans.ts`:
   - TRIAL: 100 контактов, 200 писем/мес, 1 отправитель
   - START (7 999 ₽/мес): 2 000 контактов, 5 000 писем/мес, 2 отправителя
   - PRO (19 999 ₽/мес): 10 000 контактов, 30 000 писем/мес, 5 отправителей
   Гейтинг при загрузке контактов / создании кампании / добавлении отправителя.
3. **Оплата (ЮMoney-совместимый вебхук)**: модель `Payment`, `/api/payments/webhook`
   с секретом; страница `/app/billing` (тариф, статус, «оплатить» — заглушка ссылки,
   после реальной интеграции меняется только URL). Подтверждение оплаты →
   автоактивация плана на 30 дней.
4. **Админка `/app/admin`**: клиенты (список/создание/смена тарифа), заявки с лендинга
   (+ экспорт), сводная статистика. Доступ только role=ADMIN.
5. **Оферта**: страница `/terms`, чекбокс при регистрации (обязателен),
   `User.acceptedTermsAt`. Ссылка в футере и на billing.
6. **Фиксы безопасности**: R1–R6.
7. **CI/CD**: план ниже.

## 6. CI/CD — процесс

**Цель: плохой код не попадает на прод, а плохой деплой не роняет сервис.**

1. **Локально перед пушем**: `npm run check` (tsc + build) — добавлен скрипт.
2. **GitHub Actions** (`.github/workflows/ci.yml`): на каждый push/PR — чистый
   Postgres → `prisma db push` → `next build`. Красный крестик до прода.
3. **Amvera** собирает Docker-образ из main. Если сборка упала — прод остаётся
   на старом образе (Amvera не подменяет работающий контейнер битым).
4. **Старт контейнера** (`start.sh`): `prisma db push` не фатален — при сбое
   сервер поднимается на предыдущей схеме («сайт не обновился», а не «лежит»).
5. **Рекомендуемый флоу**: изменения → `npm run check` → push → CI зелёный →
   Amvera автосборка. Для рискованных изменений — ветка + PR (CI на PR).

## 8. Архитектура — принципы (для роста без переделок)

```
src/
  lib/            — чистые модули без побочных эффектов между собой
    config.ts     — ЕДИНАЯ точка чтения env (все настройки видны в одном месте)
    plans.ts      — тарифы/лимиты (чистые функции, покрыты тестами)
    rateLimit.ts  — rate limiter (in-memory; замена на Redis — в одном файле)
    auth.ts       — сессии/роли (единственный модуль с server-only)
    services/     — АДАПТЕРЫ внешних API (unisender/claude/bitrix):
                    mock без ключа ↔ live с ключом, интерфейс не меняется
  server/         — доменная логика (вызывается из actions, API и worker)
    sendEngine    — отправка (очередь на БД, throttle, warm-up, suppression)
    inboundEngine — ответы → AI-диалог → квалификация → CRM
    billing.ts    — платежи/активация планов (шлюз-агностик)
    limits.ts     — гейтинг тарифов (одно место, не размазано по actions)
  app/            — тонкие страницы/actions: parse → вызов server/* → render
```

Ключевые решения для масштабирования:
1. **Смена платёжного шлюза** = правка только парсинга в webhook route; `billing.ts` не трогается.
2. **Смена email-провайдера** = новый адаптер в `services/` с тем же интерфейсом `sendEmail()`.
3. **Очередь**: сейчас на Postgres (достаточно до ~десятков тысяч писем/день). Путь роста:воркер уже отдельный процесс → замена опроса БД на Redis/BullMQ локализована в `sendEngine`+`worker`.
4. **Rate limiter**: in-memory (1 инстанс). При горизонтальном масштабировании — Redis, интерфейс `rateLimit()` сохраняется.
5. **Мультиарендность**: все выборки ЛК фильтруются по `userId`; админ-операции — через `requireAdmin`.
6. **Тарифы**: добавление плана = одна запись в `PLANS` + значение enum в схеме.
