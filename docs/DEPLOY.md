# Деплой Smailee на Amvera

Пошаговая инструкция. Первый деплой — ~40–60 минут.

> **Зачем вообще прод:** домашний интернет-провайдер режет исходящий SMTP-порт 465,
> поэтому локально отправка писем не работает (приём по IMAP — работает).
> На сервере порты обычно открыты. **Это первое, что нужно проверить** (шаг 6).

---

## Что деплоится

Один контейнер, внутри **два процесса** (см. `start.sh`):

| Процесс | Что делает | Без него |
|---|---|---|
| Next.js (`server.js`) | лендинг + кабинет | нет сайта |
| worker (`dist/worker.js`) | отправка, IMAP-приём, ИИ-диалог, прогрев, здоровье флота | **сайт есть, но рассылка не работает** |

Amvera не поддерживает docker-compose (один Dockerfile = один контейнер), поэтому
оба процесса живут в одном контейнере. Падение любого = перезапуск контейнера.

---

## Шаг 1. Создать проект в Amvera

1. Зарегистрируйся на [cloud.amvera.ru](https://cloud.amvera.ru).
2. Создай приложение: тип — **Docker** (у нас есть `Dockerfile` и `amvera.yml`).
3. Тариф: для теста хватит минимального. Учти — контейнер должен работать
   постоянно (воркер), а не «просыпаться по запросу».

## Шаг 2. Создать PostgreSQL

1. В Amvera создай сервис **PostgreSQL** (управляемый кластер).
2. Задай свои имя пользователя, пароль, имя БД (дефолтные зарезервированы).
3. Строка подключения изнутри Amvera:
   ```
   postgresql://<user>:<password>@amvera-<username>-cnpg-<db-project>-rw/<dbname>
   ```
   Где `-rw` — endpoint на запись (есть ещё `-ro` только чтение, `-r` любой узел).
4. **SSL обязателен** — клиент должен его поддерживать. Если Prisma будет ругаться
   на SSL, добавь в конец строки: `?sslmode=require`.

## Шаг 3. Переменные окружения и секреты

В проекте Amvera → раздел **Переменные** → «Добавить переменную или секрет».

⚠️ **Переменные недоступны на этапе сборки** — только в рантайме. Наша сборка это
учитывает (проверено: `npm run build` проходит без `.env`).

**Секреты** (хранятся в защищённом хранилище — используй именно их):

| Имя | Значение |
|---|---|
| `DATABASE_URL` | строка подключения из шага 2 |
| `JWT_SECRET` | длинная случайная строка (`openssl rand -hex 32`) |
| `MAILBOX_ENC_KEY` | **32 байта hex** (`openssl rand -hex 32`) — ключ шифрования SMTP/IMAP-паролей |
| `DEEPSEEK_API_KEY` | ключ DeepSeek (без него ИИ работает в mock-режиме) |

**Обычные переменные:**

| Имя | Значение |
|---|---|
| `APP_URL` | публичный адрес приложения (`https://<проект>.amvera.io`) |
| `ADMIN_EMAIL` | email, который получит роль ADMIN при регистрации |
| `NODE_ENV` | `production` |

**Опционально:**

| Имя | Когда нужно |
|---|---|
| `BITRIX24_WEBHOOK_URL` | передача лидов в CRM |
| `SETUP_NOTIFY_EMAIL` | куда слать заявки «Настройте за меня» (дефолт `zayt_eg@mail.ru`) |
| `WARMUP_DAY_MS` | **тест-режим прогрева**: `60000` = 1 мин вместо суток. На боевом **не задавать** |

> 🔑 `MAILBOX_ENC_KEY` на проде должен быть **свой**, не тот же, что локально.
> Ящики, подключённые с одним ключом, не расшифруются с другим — их придётся
> переподключить.

## Шаг 4. Привязать git и запушить

```bash
# один раз
git remote add amvera https://git.amvera.ru/<username>/<имя-проекта>

# каждый деплой (наша ветка main → их master)
git push amvera main:master
```

Сборка стартует автоматически при пуше в `master`. Логи: проект → **Лог сборки**.

> Если пушить `git push amvera master` — будет ошибка, т.к. у нас ветка `main`.
> Нужно именно `main:master`.

## Шаг 5. Проверить, что поднялось

Открой **Лог приложения**. Должно быть:

```
[startup] prisma migrate deploy...
[startup] OK: миграции применены
[startup] запускаю Next.js на порту 3000...
[startup] запускаю worker...
[startup] оба процесса запущены (web=... worker=...)
[worker] Smailee worker запущен (M2: пул ящиков; M3: IMAP-приём + AI-диалог; M4: прогрев)
```

Если видишь `FATAL: prisma migrate deploy упал` — проверь `DATABASE_URL` и SSL.
Приложение намеренно не стартует на неприменённой схеме (данные при этом целы).

## Шаг 6. 🎯 Главная проверка — SMTP-порты

**Ради этого весь деплой.** В документации Amvera нет подтверждения, что
исходящие 465/587 открыты (порт 25 режут почти все облака). Проверяем эмпирически:

1. Открой кабинет на проде → **Инфраструктура** → подключи ящик Яндекса.
2. Смотри статус:
   - **«Подключён»** → 🎉 порты открыты, отправка работает, можно тестировать всё.
   - **«Недоступен»** + таймаут SMTP → Amvera тоже режет 465. Пиши в поддержку
     Amvera с просьбой открыть исходящий 465/587, либо ищи другой хостинг.

## Шаг 7. Проверить прогрев

Нужно **≥2 подключённых ящика** (или ящик + seed). В логе приложения через
несколько минут появится:
```
[worker] warmup send: sent=N failed=0
```
События видны в админке → **Флот прогрева** → «Последние события прогрева».

Для быстрого теста поставь `WARMUP_DAY_MS=60000` — прогрев дойдёт до `warm` за
~14 минут вместо 14 дней. **Перед боевым использованием переменную убрать.**

## Обновление приложения

```bash
git push amvera main:master
```

Или через CI/CD (см. ниже) — тогда просто `git push origin main`.

### Что происходит с данными при обновлении

`start.sh` запускает `prisma migrate deploy` — он применяет **только новые
миграции** и **никогда не удаляет данные** (в отличие от `db push --accept-data-loss`).
Если миграция не применилась — контейнер не стартует, данные целы, в логе ошибка.

### Изменил схему — создай миграцию

```bash
# 1. правишь prisma/schema.prisma
# 2. создаёшь миграцию (применится к локальной БД)
npx prisma migrate dev --name понятное_имя

# 3. коммитишь prisma/migrations/ вместе с кодом
git add prisma/migrations prisma/schema.prisma
git commit -m "..."
```

⚠️ **`prisma db push` больше не использовать** — он не оставляет следа в
`prisma/migrations/`, и прод не узнает об изменении схемы. CI это ловит:
шаг «Схема и миграции синхронны?» покраснеет.

### Проверить миграцию на ЧИСТОЙ базе, до пуша

```bash
npm run db:migration-check
```

Пересоздаёт локально пустую БД `smailee_migration_check`, прогоняет на ней
`prisma migrate deploy` с нуля и удаляет БД. Ловит именно тот класс багов,
который не виден на твоей рабочей локальной БД (она может отличаться от
чистой прод-схемы: например, констрейнт, добавленный когда-то руками) — так
однажды упал прод в другом проекте: `ON CONFLICT (a, b)` в миграции без
уникального констрейнта на `(a, b)` — локально работало, на чистой схеме падало.

Гоняй **вручную**, когда меняешь `prisma/schema.prisma` или добавляешь
миграцию — в обычный `npm run check` / pre-push хук это не входит (полный
цикл дольше, чем пуш без изменений схемы).

### Бэкап перед рискованной миграцией

Управляемый Postgres в Amvera умеет бэкапы — сделай снапшот перед миграцией,
которая удаляет/переименовывает колонки.

## Локальный pre-push хук

`.githooks/pre-push` гоняет `npm run check` (типы + smoke + сборка) **перед
каждым `git push`** и блокирует пуш, если что-то красное. Активируется сам
через `postinstall` при `npm install` (переживает переклонирование). Осознанно
обойти: `git push --no-verify`.

---

## CI/CD

### `ci.yml` — проверка на каждый push/PR в `main`
1. Чистый Postgres 16 (как прод)
2. `tsc` — типы
3. `npm run smoke` — тесты чистых функций
4. `prisma migrate deploy` — те же миграции, что пойдут на прод
5. **Проверка рассинхрона** схемы и миграций
6. `npm run build` — Next.js standalone + бандл воркера

Не гейтит деплой (см. ниже) — просто красный крестик на GitHub, если что-то сломано.

### Как на самом деле катится код

Amvera подключена к GitHub **напрямую** (вебхук на push в `main`) и деплоит
сама — без участия GitHub Actions. Значит:
- локальный **pre-push хук** — единственный гейт перед тем, как код вообще
  уйдёт из твоей машины;
- `ci.yml` — независимая проверка постфактум (полезна, если хук обошли
  через `--no-verify`, или просто как второй взгляд);
- откат при поломке — `git revert <коммит> && git push` (Amvera задеплоит
  отменённый код по тому же вебхуку). Для отката миграции — см. бэкап выше.

### `deploy.yml` — отключён от автозапуска, только вручную

Раньше пушил `main → master` в Amvera автоматически после зелёного `ci.yml`
— альтернативный путь, если бы Amvera НЕ была подключена к GitHub напрямую.
Сейчас оставлен только для ручного запуска (`workflow_dispatch`) — пересобрать
в Amvera без нового коммита. Если понадобится вернуть автогейт через CI —
нужно **выбрать что-то одно**: либо прямой вебхук Amvera↔GitHub, либо
`deploy.yml` по `workflow_run`, не оба сразу (иначе двойной деплой).

Секреты для ручного запуска (Settings → Secrets and variables → Actions):

| Секрет | Значение |
|---|---|
| `AMVERA_USERNAME` | логин Amvera |
| `AMVERA_PASSWORD` | пароль Amvera |
| `AMVERA_PROJECT` | имя проекта (транслитом, как в адресе git-репозитория) |

> ⚠️ Чтобы workflow-файлы попали в GitHub, у токена должен быть scope **`workflow`**.
> Без него push с `.github/workflows/` отклоняется.

## Известные ограничения

- **SMTP-порты Amvera не подтверждены документально** — проверяются шагом 6.
- **Валидация ящика синхронная**: при подключении делается реальный коннект к
  SMTP+IMAP (таймаут 12 с). Импорт большого CSV будет медленным.
- **Один контейнер = web + worker.** Если понадобится масштабировать их отдельно —
  разносим на два приложения Amvera (общая БД, воркер запускать с `dist/worker.js`).
- **Прогрев требует ≥2 ящика** в сети (или seed-пул, §5.6, §9.1).
