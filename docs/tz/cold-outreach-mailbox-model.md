## 1. Контекст и стратегические решения (почему так)

### 1.1 Что за продукт
Smailee — **AI-сотрудник для холодного email-аутрича**. Ключевая ценность (УТП):
не «отправить письма», а **AI сам ведёт диалог с ответившими и отдаёт готовых
квалифицированных лидов** (см. `PRODUCT.md`, F9/F10). Отправка — commodity;
диалог + квалификация ИИ — то, чем отличаемся от инструментов вроде Trigga,
которые отдают человеку «инбокс», а не готовых лидов.

### 1.2 Почему НЕ Unisender / транзакционный ESP (отвергнуто)
- Транзакционные ESP **не принимают входящие письма** — без приёма ответов диалог
  невозможен в принципе (ядро продукта).
- Их ToS, как правило, **запрещают холодные рассылки**.
- Модель «личное письмо от Ивана, на которое отвечают» — это **реальный почтовый
  ящик** (SMTP+IMAP), а не fire-and-forget API.
- **Вывод:** Unisender из продукта **удаляется целиком** (см. §7).

### 1.3 Почему НЕ отправка с домена клиента напрямую (отвергнуто)
- Отправка холодной базы с **основного домена компании** сжигает его репутацию за
  2–3 месяца → у клиента в спам уходит настоящая деловая почта.
- Поддомен основного домена изолирует репутацию лишь **частично** (блок-листы
  листят на уровне регистрируемого домена).
- **Вывод:** используются **отдельные нейтральные домены** с названием компании
  (напр. `companytech.ru`), не основной домен клиента.

### 1.4 Физика доставляемости (жёсткие правила — законы, не пожелания)
- **≤ 30 холодных писем/день с ЯЩИКА** (`логин@домен`).
- **≤ 120 холодных писем/день с ДОМЕНА** (сумма по ящикам домена).
- **≤ 3–4 ящика на домен** (следствие: 4 × 30 = 120).
- Объём требует **флота ящиков**: ~10 на 2 000 получателей, ~50 на 10 000, ~100 на
  20 000. Одним ящиком/доменом объём не закрыть.
- Провайдер ящиков — **Яндекс 360 или Google Workspace** (высокая доставляемость в
  РФ). Основной выбор для РФ — Яндекс 360.
> Прогрев считается **отдельным потоком** и в лимит 30 не входит (§5.6, §1.7).

### 1.5 Выбранная модель — «Модель C»: клиент приносит свои ящики
Из трёх моделей (A — мы автоматизируем провижининг на аккаунтах клиента через
OAuth; B — всё на нашем аккаунте; C — клиент сам поднимает инфраструктуру и отдаёт
IMAP/SMTP-доступы) выбрана **C**:
- **Ответственность на клиенте.** Клиент — владелец доменов/аккаунтов и **автор
  рассылки**. Холодный аутрич в РФ юридически «серый» (ст. 18 «О рекламе»,
  152-ФЗ); при иске отвечает автор — т.е. клиент. Инфраструктура на клиенте делает
  эту позицию сильной.
- **Изоляция при блокировке.** Аккаунты у каждого клиента свои → блок одного не
  роняет остальных; клиента можно мигрировать на fallback-провайдер.
- **Мы не держим чужие мощные токены** (в отличие от A).
- Это же модель Trigga: провижининг не автоматизируют, оркеструют только
  отправку/прогрев/инбокс поверх ящиков клиента.

### 1.6 Сквозные принципы архитектуры (обязательны)
1. **Провайдер-агностичность.** Ядро (отправка/приём/оркестрация) работает с
   абстракцией «ящик с IMAP/SMTP-доступами», НЕ завязано на Яндекс. Провайдер —
   параметр (профиль host/port/security), чтобы иметь fallback. На старте
   реализован **единственный профиль — Яндекс 360**, но код к нему не привязан.
2. **Пер-клиентская изоляция.** Ящики, домены, лимиты, suppression каждого клиента
   изолированы. Никакого глобального ключа/аккаунта на всех.
3. **Клиент — владелец и автор.** Провижининг инфраструктуры вне системы (на
   стороне клиента по нашему гайду). Система подключается к готовым ящикам.

### 1.7 Принцип стоимости ИИ (два уровня)
- **Высокочастотное, дешёвое-за-штуку** (контент прогрева, уникализация писем) →
  **детерминированно или батч-раз**. Никаких per-item LLM-вызовов.
- **Низкочастотное, ценное** (ИИ-диалог с реально ответившими лидами, генерация
  базового текста письма) → LLM оправдан, объём естественно ограничен.
> Практический результат: прогрев — 0 ИИ; отправка кампании — 0 ИИ в рантайме
> (уникальность детерминирована); LLM тратится только на диалог с ответившими и на
> разовую сборку вариантов письма. См. §5.9.

(DNS/DKIM/Project) для модели C не нужна.

## 3. Глоссарий

- **Ящик (Mailbox)** — единица отправки/приёма: `логин@домен` с доступами
  IMAP+SMTP. В коде заменяет прежний `Sender`.
- **Домен-группа (DomainGroup)** — все ящики на одном домене; на неё лимит 120/день.
- **Персона / алиас** — контактное лицо («Иван Иванов»), от имени которого идёт
  переписка; 2–3 алиаса (`i.ivanov`, `ivanov.i`, …). В Я360 один «сотрудник» × N
  доменов = N ящиков.
- **Прогрев (warmup)** — авто-имитация живой переписки между ящиками для роста
  репутации. Обязателен 14 дней до рассылки, не выключается никогда.
- **Seed-пул** — наши собственные ящики в сети прогрева, чтобы сеть работала с
  первого клиента.
- **Пиры прогрева** — ящики, с которыми прогревается данный: seed + ящики других
  клиентов (кросс-клиентский прогрев) + свои «соседи».
- **Скрытый код прогрева** — неприметная метка в теле прогревочного письма для
  распознавания и скрытия из инбокса (НЕ заголовок вроде `X-Warmup`).
- **Движок уникальности (spintax)** — детерминированная подсистема, делающая каждое
  письмо уникальным ротацией фраз/форм + подстановкой переменных. Без ИИ в рантайме.
- **Корпус прогрева** — рукописный сборник диалогов (со spintax-разметкой), по
  которому алгоритм строит прогревочную переписку.

## 4. Модель данных (словами, не кодом)

### 4.1 Mailbox (заменяет `Sender`)
Провайдер-агностичный ящик клиента. Поля:
- владелец — ссылка на `User` (клиент);
- email-адрес; отображаемое имя отправителя (Sender Name);
- провайдер — enum (`yandex` | `google` | `other` | `custom`) — для UI/подсказок,
  НЕ для ветвления логики;
- SMTP: host, port, security (SSL/STARTTLS), login;
- IMAP: host, port, security, login;
- доступы (пароли SMTP/IMAP) — **хранить зашифрованно** (§8.2); в модели ссылка на
  зашифрованное значение, не сам пароль;
- ссылка на домен-группу (DomainGroup);
- **счётчик холодной отправки:** дневной лимит (default 30) + отправлено сегодня +
  дата счётчика;
- **счётчик прогрева (ОТДЕЛЬНЫЙ):** отправлено прогревом сегодня + дата + текущая
  ступень ramp (день прогрева);
- состояние прогрева — enum (`off` | `warming` | `warm`) + дата старта;
- состояние подключения — enum (`ok` | `auth_error` | `unreachable` | `paused`);
- здоровье/скоринг + дата последней проверки;
- когда последний раз валидировался.

### 4.2 DomainGroup
- владелец (User); имя домена;
- дневной лимит домена (default 120); отправлено холодным сегодня + дата.
> Нужна, т.к. лимит 120/день — на домен, а ящиков на домене несколько. Группа
> выводится из email-домена ящика при подключении.

### 4.3 Изменения в Message
- ссылка на **Mailbox**, через который письмо отправлено (для непрерывности треда
  при ответе и учёта лимитов);
- поля связывания входящего ответа с исходящим (Message-ID / In-Reply-To /
  References / адрес треда) — фундамент привязки диалога.

### 4.4 WarmupEvent (новое)
Учёт прогревочной переписки: ящик-отправитель, ящик-получатель, скрытый код,
статус (`sent` | `delivered` | `opened` | `replied` | `rescued_from_spam`), время.
Нужен для контроля ramp, здоровья и отделения прогрева от реального инбокса.

### 4.5 Корпус прогрева (данные, §5.9.3)
Рукописный набор диалоговых деревьев со spintax-разметкой. Формат — в §5.9.3.

## 5. Подсистемы и требования

### 5.1 Подключение и валидация ящиков
- Импорт пула через **CSV** (email, Sender Name, SMTP-пароль, IMAP-пароль;
  host/port подставляются по провайдеру) + ручное добавление в UI.
- По каждому ящику при подключении: **тест IMAP-логина + тест SMTP-отправки** →
  проставить состояние подключения; ошибочные подсветить с причиной.
- Автоматически определить домен и привязать/создать DomainGroup.
- Предупреждать, если на домене >4 ящиков.

### 5.2 План-калькулятор инфраструктуры
- Вход: целевой объём получателей/мес.
- Выход: сколько доменов, персон, ящиков (по правилам 30/120/≤4); схема («3 домена
  × 4 персоны = 12 ящиков»), подсказки имён доменов (нейтральные, с названием
  компании, без цифр/дефисов) и по 3 алиаса на персону.
- Часть гайда/онбординга: система считает, клиент исполняет руками.

### 5.3 Оркестрация отправки (ядро, заменяет `sendEngine`)
- Кампания раскидывается **по пулу ящиков клиента**, не через один канал.
- Соблюдать: **≤30 холодных/день на ящик**, **≤120/день на домен**, троттлинг,
  растягивание по дням, **ротация персон и доменов**.
- Отправка каждого письма — через **SMTP конкретного ящика** (транспорт
  провайдер-агностичен).
- Упёрлись в дневной лимит → письма остаются в очереди до следующего дня
  (resumable).
- Каждое письмо **проходит через движок уникальности (§5.9)** перед отправкой.
- Прогрев идёт **параллельно, отдельным счётчиком**, не входит в лимит 30 (§5.6).

### 5.4 Приём ответов (ядро, заменяет вебхук `inboundEngine`)
- **IMAP-поллинг** каждого ящика по расписанию.
- Новое письмо → определить: прогревочное (по скрытому коду → в WarmupEvent,
  скрыть) или реальный ответ лида.
- Реальный ответ → **связать с исходным Message/Contact/Campaign** → передать в
  ИИ-слой (§5.5).

### 5.5 ИИ-диалог и квалификация (УТП, переиспользует `llm.ts`)
- По входящему ответу ИИ генерит ответ и шлёт его **через тот же ящик** (треды/
  репутация).
- ИИ **квалифицирует** лида (тёплый/холодный/нецелевой).
- На пилотах — режим **модерации**: ответ ИИ показывается оператору до отправки.
- Это единственный высокочастотный LLM-поток, но объём ограничен числом
  ответивших (§1.7, tier 2).

### 5.6 Движок прогрева
**Модель — гибрид, старт на своём seed-пуле, кросс-клиентский пиринг.**

- **Взаимная сеть.** Ящики всех клиентов с включённым прогревом + seed-пул
  переписываются **между собой** (кросс-клиентский пиринг). Сеть растёт и
  оздоравливается с каждым клиентом. Служебный характер трафика фиксируется в
  оферте.
- **Seed-пул.** ~20 наших ящиков на ~3 доменах (Я360), заранее прогретых. Критичен
  только для первых 1–2 клиентов (обеспечивает разнообразие пиров); к 3–4-му
  клиенту сеть самоподдерживается. Операционка: ~2 000₽/мес, поднимает и держит
  оператор.
- **Выбор пиров.** Каждый прогреваемый ящик обменивается с **ротируемым
  разнообразным подмножеством** пула (не одни и те же партнёры), из seed + ящиков
  других клиентов + своих «соседей».
- **Отдельный счётчик** (§4.1), в лимит 30 холодных НЕ входит.
- **Ramp:** день 1 — ~2–4 письма, +2–4/день, к дню 14 — ~20–30/день, дальше
  поддерживающий уровень. **14 дней обязательны до старта кампаний. Не
  выключается никогда**, в т.ч. во время кампаний.
- **Поведение принимающей стороны** (мы контролируем IMAP обоих концов — seed и
  клиента): отметить прочитанным; ответить (~30–50%, иногда многоходовой тред);
  **вытащить из спама** — IMAP-перемещение Спам→Входящие (сигнал «не спам» для
  Яндекса); иногда пометить важным.
- **Тайминг рандомизирован** (не ровно N в фиксированные часы).
- **Маркер** — неприметный код в теле (не заголовок).
- **Контент — рукописный корпус + spintax (§5.9), 0 ИИ.**

> Риск и зависимость: без прогрева флот бесполезен (сразу спам). Seed-пул нужен
> рано — служит и тестовой средой для отладки отправки/приёма (M1–M3).

### 5.7 Единый инбокс
- Только **реальные ответы** людей; прогрев автоматически скрыт.
- Карточка диалога (email-тред), статус квалификации, действия оператора.

### 5.8 Здоровье и мониторинг флота
- Отслеживать скоринг/здоровье, всплески отказов/жалоб/bounce.
- Авто-пауза ящика при просадке; вывод «выгоревших» из ротации.
- Дашборд состояния флота для оператора.

### 5.9 Движок уникальности (spintax + переменные)
Standalone-подсистема, делающая **каждое** письмо уникальным. **Детерминированная,
без ИИ в рантайме.** Общая для прогрева и боевых рассылок (требование уникальности
Яндекса одинаково для обоих).

**5.9.1 Синтаксис.**
- Альтернативы: `{привет|здравствуйте|добрый день}`, с вложенностью.
- Переменные: `{{name}}`, `{{company}}` и т.п. (расширяет существующий `render()`
  в `sendEngine`).
- Send-time: алгоритм детерминированно собирает **уникальный вариант** на каждое
  письмо. **0 ИИ, 0 стоимости за письмо.** Обеспечить высокую вариативность между
  получателями (избегать коллизий).

**5.9.2 Сборка вариантов (build-time, только кампании, опционально).**
- Дешёвый **батч**-вызов LLM превращает базовый текст письма в spintax-разметку
  (синонимы фраз/конструкций). **Короткий системный промпт, короткий output,
  сгруппировано. Один раз на кампанию/шаблон, не на письмо** (§1.7, tier 1).

**5.9.3 Корпус прогрева (формат данных).**
- Рукописный (оператором) сборник **диалоговых деревьев**: открывающие реплики →
  ветки ответов → продолжения; каждый узел размечен spintax; тематика — безобидный
  деловой small-talk.
- «Навигация алгоритмами» — детерминированный/рандомный обход (выбрать открытие →
  ветку → продолжение). **ИИ не участвует ни в рантайме, ни на этапе сборки.**
- Формат должен позволять просто «дописывать» корпус руками (напр. структурированные
  файлы), без изменения кода.

## 6. Жизненный цикл (сводка)

| Фаза | Клиент | Оператор | Система |
|---|---|---|---|
| 0. Заявка → кабинет | оставляет заявку | заводит кабинет | LandingLead→User (есть) |
| 1. Инфраструктура | покупает домены, поднимает Я360, DNS, персоны, IMAP | гайд / «под ключ» | план-калькулятор (5.2), чек-лист |
| 2. Подключение ящиков | грузит CSV доступов | проверяет | валидация, Mailbox+DomainGroup (5.1) |

## 7. Что делаем с существующим кодом (явно)

### Оставляем почти как есть
Лендинг/заявки, кабинет/онбординг, `Contact`, `Campaign`, ИИ-генерация/ответ/
квалификация (`src/lib/services/llm.ts`), `Lead`, дашборд, follow-up, suppression,
Битрикс, уведомления о тёплом лиде (F14).

### Переделываем серьёзно
- **`Sender` → `Mailbox`** (§4.1): «ящик с IMAP/SMTP-доступами», отдельные счётчики
  холодных и прогрева.
- **`src/server/sendEngine.ts`**: → **оркестрация пула через SMTP** с лимитами
  30/120, ротацией и прогоном через движок уникальности (§5.3, §5.9).
- **`src/server/inboundEngine.ts`**: → **IMAP-поллинг** ящиков (§5.4).
- **`render()`** в sendEngine → расширить до полноценного движка уникальности (§5.9).

### Удаляем (полностью)
- Адаптер `src/lib/services/unisender.ts`, `User.unisenderApiKey`,
  managed-поддомены (`MAIL_BASE_DOMAIN`, slug `<client>.send.smailee.ru`),
  верификация домена через Unisender, вебхук `/api/inbound`.
- Контент-маркетинговая opt-in фича (серии писем), если завязана на Unisender —
  удаляется вместе с ним.

> Исполнителю: не «чинить» старую Unisender-ветку под аутрич — она архитектурно не
> подходит (§1.2). Двигаемся на модель ящиков.

## 8. Нефункциональные требования и ограничения

### 8.1 Провайдер-агностичность
Транспорт (SMTP-отправка, IMAP-приём) абстрагирован интерфейсом. Провайдер =
профиль host/port/security. На старте реализован **только профиль Яндекс 360**, но
движки к нему не привязаны; добавление Google/др. = новый профиль без изменения
движков.

### 8.2 Безопасность доступов
Пароли IMAP/SMTP клиента — чувствительные. Хранить **зашифрованно** (шифр на уровне
приложения, ключ в env/секрете, не в БД открыто). В логах не светить. Доступ к
расшифровке — только движки отправки/приёма.

### 8.3 RU-специфика доставляемости
- Обязательная уникальность письма — через движок уникальности (§5.9).
- Лимиты 30/120 — на уровне движка, не «на совесть».
- Яндекс привередлив к уникальности (неуникальные письма → ошибки → потеря лидов).

### 8.4 Юридический слой (не код, но обязательное условие)
- Оферта фиксирует: **клиент — отправитель, автор рассылки, оператор ПДн**;
  отвечает за согласия и легальность базы; indemnity.
- Служебный характер прогревочного трафика (кросс-клиентский обмен) — раскрыт в
  оферте.
- Инфраструктура оформлена на клиента (это и переносит ответственность).

### 8.5 Стоимость ИИ
См. §1.7 (два уровня). Высокочастотное — детерминированно/батч; per-item LLM
только в диалоге с ответившими.

## 9. Открытые вопросы и отложенное

### 9.1 Bootstrapping и операционка seed-пула
Модель прогрева спроектирована (§5.6). Остаётся операционная задача: поднять ~20
seed-ящиков, прогреть их, держать здоровыми. Спроектировать процесс обслуживания
seed-пула (мониторинг, замена выгоревших).

### 9.2 Включение IMAP/app-пароля
В модели C — на стороне клиента (по гайду). Если позже пойдём в модель A
(автопровижининг), проверить эмпирически, доступно ли включение IMAP и генерация
app-пароля через API Яндекс 360 (по докам — не подтверждено).

### 9.3 Ценообразование vs стоимость инфраструктуры
При 20k объёме инфраструктура (~100 ящиков) ≈ 8–9к₽/мес на клиента, что превышает
базовый тариф 7 999₽ (PRODUCT.md). Пересчитать: инфра поверх подписки (как Trigga)
либо лимиты объёма по тарифам. Влияет на продукт, не на код.

## 10. Рекомендуемые этапы внедрения (milestones)

Одно ТЗ, реализуется инкрементально:

- **M1 — Фундамент ящиков.** Модель `Mailbox` + `DomainGroup` (§4), подключение и
  валидация (CSV+UI, §5.1), план-калькулятор (§5.2). Миграция `Sender`→`Mailbox`,
  удаление Unisender-ядра (§7).
- **M1.5 — Движок уникальности (§5.9).** Кросс-каттинг: нужен и для отправки (M2),
  и для прогрева (M4). Синтаксис spintax + переменные + send-time рендер. (Сборку
  вариантов build-time можно добавить чуть позже.)
- **M2 — Оркестрация отправки.** Пул, лимиты 30/120, ротация, throttle, расписание,
  прогон через §5.9. Переписанный `sendEngine`.
- **M3 — Приём + ИИ-диалог.** IMAP-поллинг (§5.4), привязка треда, ИИ-ответ через
  тот же ящик, квалификация, модерация (§5.5). Переписанный `inboundEngine`.
- **M4 — Прогрев.** Движок прогрева + seed-пул + корпус + скрытый код + spam-rescue
  (§5.6). Использует §5.9 и seed-ящики из ранних этапов.
- **M5 — Инбокс, здоровье, дашборд флота** (§5.7, §5.8).

Каждый milestone — самостоятельно проверяемый. M1 + M1.5 — фундамент; с них начинать.

## 11. Статус решений

**Приняты (зафиксированы в этой версии):**
- Модель C (клиент приносит ящики, мы оркеструем + ИИ-диалог).
- Unisender и контент-фича — **удаляются** из продукта.
- Провайдер-агностичность; на старте — только профиль Яндекс 360.
- Прогрев: гибрид, старт на своём seed-пуле (~20 ящиков), **кросс-клиентский**
  пиринг, **отдельный счётчик** от холодных, ramp 14 дней, не выключается.
- Уникальность: детерминированный spintax-движок, общий для прогрева и рассылок;
  контент прогрева — рукописный корпус (0 ИИ); build-time сборка вариантов
  кампаний — дешёвый батч-LLM.

**Осталось прояснить (не блокирует M1):**
- Операционка seed-пула (§9.1), ценообразование (§9.3).
