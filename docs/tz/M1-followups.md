# M1–M3 — что осталось ввести/доделать (follow-ups)

Список того, что сделано заглушкой/не проверено вживую и требует реальных
данных или доработки. Общая причина по всем пунктам одна: **нет живых
app-паролей Яндекс 360** для тестового ящика в текущей среде.

## 1. Реальная валидация ящиков при подключении (§5.1) — сейчас ЗАГЛУШКА
`src/lib/mail/validate.ts` (вызывается при подключении ящика в UI) пока делает
только структурную проверку (формат, наличие host/port/логина/паролей), без
сетевых коннектов — хотя сами SMTP/IMAP-клиенты (`nodemailer`, `imapflow`) уже
подключены и реально используются в M2 (`src/lib/mail/transport.ts`,
`verifySmtpAuth`) и M3 (`src/lib/mail/imap.ts`, `pollMailboxInbox`).

**Что понадобится:**
- **App-пароли Яндекс 360** для тестовых ящиков (не пароль от аккаунта):
  включить IMAP в настройках Яндекс 360 и создать пароль приложения.
- Переключить `validateMailbox` на реальные вызовы: `verifySmtpAuth` (уже
  есть, M2) + аналогичный IMAP-login-тест (M3: можно переиспользовать первый
  шаг `pollMailboxInbox` — успешное подключение+SELECT само по себе валидирует
  IMAP-логин).
- Проверить в среде прод/дев, что исходящие порты 465 (SMTP) и 993 (IMAP)
  открыты (в частности из контейнера Amvera).

## 2. Ключ шифрования доступов
- Задать `MAILBOX_ENC_KEY` в `.env` (dev) и в env Amvera (прод):
  `openssl rand -hex 32`. Без него подключение ящиков не сохранит пароли.
  (Уже сделано локально в этой сессии.)

## 3. M2/M3 реально отправляют/принимают почту, но НЕ проверены на живом Яндексе
- **M2** (`sendEngine.ts`, `src/lib/mail/transport.ts`): протестирован реальным
  вызовом `nodemailer` против самодельного фейкового SMTP-сервера (ротация,
  лимиты 30/120, resumable-очередь — всё подтверждено). Против настоящего
  Яндекса не гонялся.
- **M3** (`inboundEngine.ts`, `src/lib/mail/imap.ts`): ИИ-диалог + модерация +
  реальная отправка ответа с корректными `In-Reply-To`/`References` —
  протестированы полностью (реальный DeepSeek + фейковый SMTP). Обработка
  ошибок IMAP-подключения (auth/network, троттлинг по `lastCheckedAt`,
  исключение сломанных ящиков из ротации поллинга) — тоже протестирована.
  **Не протестирован** только сам факт успешного IMAP `FETCH` живого письма:
  писать корректный фейковый IMAP-сервер (полный протокол IMAP4rev1 —
  тегированные команды, литералы, UIDVALIDITY/UIDNEXT) непропорционально
  дорого ради теста; код в `imap.ts` сверен построчно с официальными типами
  `imapflow` (`node_modules/imapflow/lib/imap-flow.d.ts`). Проверить первым
  реальным ящиком.

## 4. M4 (движок прогрева, §5.6) — что проверено, а что нет
- **Проверено живьём** (фейковый SMTP + реальная dev-БД, `scripts/_test-warmup.ts`,
  удалён после прогона): `processWarmupSendRound` — автостарт `off→warming`,
  ramp по дням (день1 → warmupState=warm на дне 14), выбор пиров (seed
  всегда доступен), создание `WarmupEvent`, маркер в HTML-теле переживает
  реальный SMTP-transport и корректно вытаскивается тем же `mailparser`, что
  использует IMAP-приём (`extractWarmupCode` на распарсенном письме). Корпус
  (`src/lib/warmup/corpus.ts`) и детерминированный RNG (`src/lib/rng.ts`)
  проверены отдельно на чистой логике.
- **Не проверено** (тот же разрыв, что и в M3, п.3 выше — нет живых
  IMAP-доступов): `processWarmupEngagement` (`markSeen`/`flagImportant` через
  `imapflow.messageFlagsAdd`) и `processWarmupSpamRescue`
  (`imapflow.messageMove` из папки "Спам"). Код сверен построчно с
  `node_modules/imapflow/lib/imap-flow.d.ts`, по тому же принципу, что и
  `pollMailboxInbox` в M3. Проверить первым реальным ящиком — как и IMAP
  `FETCH` из M3.
- `Mailbox.spamFolder` захардкожен как `"Спам"` только для профиля Яндекс 360
  (`src/lib/mail/profiles.ts`) — если имя папки на реальном аккаунте другое
  (напр. локаль интерфейса не русская), спасение из спама не найдёт папку;
  `rescueWarmupFromSpam` в этом случае просто ничего не находит (не считается
  ошибкой), но стоит проверить реальное имя папки на живом ящике.
- Seed-ящики заводятся оператором вручную вне кода (§9.1) — сейчас нет UI для
  простановки `Mailbox.isSeed`; на старте это делается прямым апдейтом в БД.
