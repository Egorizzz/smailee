# Smailee

AI-агент для холодной email-лидогенерации. Пишет персональные письма, сам ведёт
диалог с ответившими и приводит квалифицированных лидов — без найма менеджера.

Стек: **Next.js 16 (App Router) + TypeScript + Tailwind + Prisma + PostgreSQL**.

---

## Быстрый старт (локально)

Требуется: Node.js 20+, PostgreSQL 16.

```bash
# 1. зависимости
npm install

# 2. настроить .env (см. .env.example)
cp .env.example .env
# как минимум задать DATABASE_URL, например:
# DATABASE_URL="postgresql://<user>@localhost:5432/smailee_dev"

# 3. создать таблицы
npm run db:push

# 4. запуск
npm run dev
# → http://localhost:3000
```

Лендинг — на `/`, кабинет — на `/app` (после регистрации на `/register`).

## Переменные окружения

| Переменная | Обязательна | Назначение |
|------------|-------------|-----------|
| `DATABASE_URL` | да | Строка подключения к Postgres |
| `JWT_SECRET` | да (прод) | Секрет для подписи сессий. Локально есть небезопасный дефолт |
| `FAL_KEY` | нет | Генерация изображений (fal.ai). Только для dev-скриптов картинок |
| `ANTHROPIC_API_KEY` | нет | Claude: генерация писем, диалог, квалификация. Без ключа — mock |
| `UNISENDER_API_KEY` | нет | Unisender Go: реальная отправка. Без ключа — mock |
| `BITRIX24_WEBHOOK_URL` | нет | Передача тёплых лидов в CRM. Без URL — mock |

**Принцип заглушек:** внешние сервисы обёрнуты в адаптеры (`src/lib/services/*`).
Пока ключ пуст — адаптер работает в mock-режиме и не ломает сценарий. Как только
ключ появится в `.env` — тот же код начинает делать реальные вызовы. Внутренняя
логика (БД, очередь отправки, статусы, дашборд) — настоящая, не заглушка.

## Архитектура

```
src/
  app/
    page.tsx                 — лендинг (Hero, Боли, Про продукт, Features, Цена, CTA)
    (auth)/                  — вход/регистрация (login, register) + server actions
    (app)/                   — кабинет (защищён requireUser)
      app/                   — обзор, онбординг, отправитель, контакты, кампании, лиды
    api/
      leads/                 — приём заявок с лендинга + экспорт в Excel
      inbound/               — вебхук входящих ответов (прод)
      track/open/[id]/       — пиксель отслеживания открытий
  lib/
    prisma.ts                — клиент БД
    auth.ts                  — сессии (bcrypt + JWT в httpOnly cookie)
    services/                — адаптеры Unisender / Claude / Bitrix (mock-aware)
  server/
    sendEngine.ts            — движок отправки (очередь на БД + throttle)
    inboundEngine.ts         — обработка ответа: AI-диалог → квалификация → лид → CRM
    worker.ts                — фоновый worker для прода (npm run worker)
```

### Как работает рассылка
1. Кампания создаётся → на каждый контакт сегмента заводится запись `Message` (PENDING).
2. Запуск → `sendEngine` берёт письма батчами, шлёт через провайдера с паузой
   (throttle), пишет статус в БД. Отправка resumable (упал — продолжит с PENDING).
3. Ответ клиента → вебхук `/api/inbound` → `inboundEngine`: сохраняет входящее,
   AI пишет ответ, квалифицирует лида, тёплого — передаёт в Битрикс24.

## Деплой на Amvera

Проект готов к контейнерному деплою: `output: "standalone"`, `Dockerfile`,
`amvera.yml`, `start.sh` (применяет схему → запускает сервер), CI на GitHub Actions.

1. Репозиторий на GitHub: `Egorizzz/smailee` (уже запушен).
2. В Amvera создать приложение из этого GitHub-репозитория + сервис **PostgreSQL**.
3. В переменных окружения приложения Amvera задать:
   - `DATABASE_URL` — из Amvera Postgres,
   - `JWT_SECRET` — длинная случайная строка,
   - `APP_URL` — публичный URL приложения (для пикселей/кликов/отписок),
   - опционально `ANTHROPIC_API_KEY`, `UNISENDER_API_KEY`, `BITRIX24_WEBHOOK_URL`.
4. Amvera соберёт образ по `Dockerfile` (см. `amvera.yml`, `containerPort: 3000`).
   На старте `start.sh` применяет схему (`prisma db push`, не фатально) и запускает
   `node server.js`.
5. (Опционально) отдельный процесс-worker: `npm run worker` — досылка больших
   рассылок и follow-up в фоне.

> CI (`.github/workflows/ci.yml`) на каждый push поднимает чистый Postgres, гоняет
> `prisma db push` и `next build` — ловит поломки до того, как Amvera тронет прод.
> Для строгих миграций позже перейти на `prisma migrate deploy`.
